# 複合インデックス

```sql
SELECT
  e.first_name, e.last_name, d.dept_no
FROM employees AS e
JOIN dept_emp  AS d IGNORE INDEX (idx_dept_emp_deptno)
ON e.emp_no = d.emp_no
WHERE d.dept_no = 'd003';
```

実行速度を速くするためのインデックスは下だけど、複合インデックスの順番がdept_no, emp_noになっている理由を教えて

CREATE INDEX idx_dept_emp_deptno_empno ON dept_emp(dept_no, emp_no);

dept_emp(dept_no, emp_no)という順番で複合インデックスを作成する理由は、クエリのWHERE句やJOIN句で「dept_noで絞り込んだ後にemp_noでアクセス」するパターンが多い場合に最適化されるからです。[^1][^2][^3]

## インデックスの順番が重要な理由

- **複合インデックスは、左側カラムから優先的に利用されます**。今回のように(dept_no, emp_no)の順なら「dept_noで絞り込んだ後、emp_noで高速に特定」したい場合に効果的です。[^2][^1]
- 順番が逆（emp_no, dept_no）の場合、WHEREやJOINの条件でdept_noのみを使っているとインデックスが効きません。最も左のカラムが必須になる「最左前方一致の原則」があるためです。[^4][^2]

## 実際のクエリへの最適化

- 今回のSQLは「WHERE d.dept_no = 'd003'」でdept_noを限定し、JOINでemp_noを特定しています。このパターンに対して(dept_no, emp_no)のインデックスが最適です。[^1][^2]
- 実際の運用で「特定部門の従業員一覧」を頻繁に取り出すなら、この順番がテーブルアクセスのコスト削減につながります。[^3]

## カーディナリティとの関係

- 一般的にカーディナリティ（値の種類の多さ）が高いカラムを前方にすることで効率が上がる場合もありますが、「実際のクエリパターンと最初の絞り込み条件」を優先することが多いです。[^3][^1]

**まとめると、クエリでWHEREやJOINで最初に利用するカラムがdept_noなので、その並びを複合インデックスの先頭にすることで効率的にアクセスできるためです**。
